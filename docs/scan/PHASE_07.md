- `program_backup.py:102-108` `_check_column_exists` — private duplicate of the same shape as `routes/workout_plan.py::column_exists` (used to check `exercise_order`/`superset_group` presence on `user_selection`). Two near-identical "does this column exist" helpers live in different modules (`utils/program_backup.py` private one, `routes/workout_plan.py` public `column_exists`/`table_exists`). **[NEW]** — WP2.4 already plans to move `column_exists`/`table_exists` into utils; when that lands, `_check_column_exists` here becomes a natural third caller to fold into the same helper (currently 3 call sites in this file alone: `:164-165`, `:423-424`). Small dedup opportunity riding along WP2.4, not a new WP.

- `program_backup.py:111-247` `create_backup(name, note, backup_type, db=None)` — **[RISK] Not atomic on the primary user-facing path.** When called without a shared `db` (the normal case — `routes/program_backup.py:100` calls `create_backup(name=name, note=note, backup_type='manual')` with no `db` argument for every manual "Create Backup" click in the UI), `should_close=True` and `commit_writes=should_close=True` (`:146-152`). Every `execute_query(..., commit=True)` call — the header INSERT (`:191`) and **each item INSERT inside the `for item in items` loop** (`:202-219`) — calls `self.connection.commit()` individually (verified in `utils/database.py:200-229`: `execute_query(commit=True)` commits after that single statement). This means a backup with N items does **N+1 separate, non-atomic commits**, not one transaction. If the process dies or raises between item commits, the `program_backups` header row is already durably committed with `item_count = N` (computed once up front at `:185` from the full source set) while `program_backup_items` may hold fewer than N rows — a header that overstates its own contents. Contrast with `restore_backup` (`:426-538`) and `create_auto_backup_before_erase`'s call into `create_backup(..., db=db)` (`:632-646`), both of which wrap everything in one `BEGIN IMMEDIATE ... db.connection.commit()` transaction. The BaseException handler at `:240-247` correctly closes/rolls back the *connection*, but that only affects the currently-open (uncommitted) statement — prior item commits already landed. **This is worth a follow-up WP**: either always commit `create_backup` as one transaction (wrap the whole body in `BEGIN IMMEDIATE`/commit like `restore_backup`) or accept-and-document the partial-backup risk. Not in the current REFACTOR_PLAN — flagging as a new candidate, low blast radius (backup creation only, not restore).

- `program_backup.py:250-319` `update_backup_metadata` — clean; single `UPDATE` statement, no atomicity concern (one write).

- `program_backup.py:375-558` `restore_backup(backup_id)` — the four-way `if has_order and has_superset / elif has_order / elif has_superset / else` INSERT branching (`:449-533`, ~85 lines) duplicates almost-identical SQL four times to handle pre-migration schemas. **[NEW]** — `create_backup` already solves the identical "optional columns" problem more concisely by building `select_cols`/`order_clause` dynamically (`:167-176`). `restore_backup` could use the same dynamic-column-list technique to collapse ~85 lines to ~25. Pure refactor, no behavior change; good WP2-style extract-method candidate (not currently listed — flag for Phase 2 backlog, sibling to WP2.2's "decompose long functions" pattern).
  - The restore is correctly atomic: single `BEGIN IMMEDIATE` at `:427`, all inserts `commit=False`, one `db.connection.commit()` at `:535`, `except sqlite3.Error: rollback` at `:536-538`. This is the contract WP2.4 must preserve byte-for-byte — confirmed no drift risk from a plan perspective, this function's transaction shape is exactly right and should be the template if `create_backup` gets fixed per the RISK above.
  - `:430-432` explicit `DELETE FROM workout_log` before `DELETE FROM user_selection`, commented as belt-and-suspenders given the FK `ON DELETE CASCADE` — confirmed real: `utils/database.py:82` sets `PRAGMA foreign_keys = ON`, so the CASCADE is actually enforced; the explicit DELETE is redundant-but-documented, not a bug.

- `program_backup.py:561-601` `delete_backup` — two separate `execute_query` calls (delete items, then delete header), each defaulting to `commit=True` → two independent commits. Comment (`:583`) acknowledges cascade would handle it; explicit delete is intentional redundancy. Minor: if the header delete fails after items delete succeeds, no orphaned-item risk (items already gone) but this is an already-accepted pattern, not flagging as new risk.

- `program_backup.py:604-646` `create_auto_backup_before_erase()` — **[RISK]/[NEW] — dead in production, by design, but the design itself is a landmine for future refactors.** This function creates a `program_backups` row (`backup_type='auto'`), named `"Pre-Erase Auto-Backup (<timestamp>)"`, and is thoroughly unit-tested (`tests/test_program_backup.py:340-420`+, 6+ dedicated test cases). Repo-wide grep confirms **zero production callers** — `app.py`'s actual `/erase-data` route (`app.py:165-232`) never calls it; it calls `utils.auto_backup.create_startup_backup()` instead (a raw SQLite file-copy to `data/auto_backup/database_<timestamp>.db`, not a `program_backups` row). `docs/program_backups.md:64` documents this explicitly and gives the reason: `/erase-data` **drops the `program_backups` and `program_backup_items` tables** as part of the full reset (`app.py:180-181`), so any row `create_auto_backup_before_erase()` wrote would be destroyed a few lines later by the `DROP TABLE` loop anyway — using the file-level snapshot instead is the only mechanism that survives a full drop-and-recreate. **This is correct as shipped, not a bug** — but it means `create_auto_backup_before_erase` and its `prune_auto_backups(keep_count=10)` pairing (`:649-684`) are pure test-only surface with no runtime caller. Per rule 8 in REFACTOR_PLAN (test references count as live), this is not a WP0 deletion candidate, but any future WP that touches `/erase-data` or the backup registry (WP2.4) must NOT "helpfully" wire this function in — doing so would create a backup row that gets dropped a few statements later, silently no-op, and would look like a regression fix but isn't. Flagging explicitly for WP2.4 executors since WP2.4 touches the same erase/backup surface area.

- `program_backup.py:687-722` `get_latest_auto_backup`, `get_active_program_count` — straightforward, single-query, no issues.
