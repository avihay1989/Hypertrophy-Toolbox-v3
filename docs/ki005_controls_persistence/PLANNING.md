## Section 0 — Requirements Brief

**Raw request** (verbatim)
> KI-005 from `docs/UI_SCENARIOS_GAP_ANALYSIS.md` §0 — "Page refresh mid-entry loses unsaved Workout Controls values (sets/RIR/weight/min/max-rep). Form fields reset to template defaults on reload." Goal: persist and restore these controls across reload.

**Problem**
A page refresh mid-entry discards unsaved Workout Controls values (weight / sets / RIR / RPE / min-rep / max-rep). Form fields reset to template defaults on reload, losing the user's in-progress input. The desired outcome is that these controls are persisted and restored across a reload. Per the approved Gate 0 decisions, persistence uses tab-scoped sessionStorage with one global set of controls per tab, and the clearing policy is: retain on a successful Add Exercise and on Clear Filters; clear the stored values and restore template defaults on Clear Plan / Clear All; and clear naturally when the tab is closed.

**Acceptance criteria**

*Finalized at Gate 0 (2026-07-12) from the owner decisions recorded below. All previously provisional (⚠️) criteria are now resolved into firm criteria.*

1. Given a user has entered values in the Workout Controls on `/workout_plan`, when the page is reloaded (hard refresh) within the same tab, then all six control fields — weight, sets, RIR, RPE, min-rep, max-rep — are restored into their form fields instead of resetting to template defaults.
   - *Clarification (owner-authorized at Gate 1, 2026-07-12 — not drift):* "restored" means **the latest valid value emitted by the field's `input` event survives reload even if focus never left the field**. Mid-entry values are captured synchronously on `input`; a commit/blur is **not** required. Criterion 1 is explicitly **not** weakened to "committed values." If the captured value is empty, malformed, non-numeric, or out of range at reload, criterion 9's fallback applies. The owner authorized it because the council (AR-4) identified a genuine contradiction between criterion 1's "mid-entry" intent and a commit-on-blur save trigger. *(This was the **first** owner-authorized Section 0 clarification after Gate 0. A **second** followed on 2026-07-13 for criterion 3 — see the Section 0 edit log under Evidence / Notes. Both are owner rulings, not drift.)*
2. Given the six control fields, when persistence and restore run, then exactly those six fields (weight, sets, RIR, RPE, min-rep, max-rep) and no others are stored and restored. *(Resolves open question 1: RPE is included.)*
3. Given the user completes a successful Add Exercise, when the plan updates, then the current/post-success displayed control values are retained (not cleared). *(Clearing policy — retain.)*
   - *Clarification (owner-authorized at Gate 1, 2026-07-13 — a ruling, not drift):* **READING A is the contract.** Owner ruling, verbatim: *"The user-entered six Workout Controls values must remain unchanged through a successful Add Exercise and survive reload afterward. 'Retain' means retain the values the user entered before Add — not merely persist whatever reset/estimate values happen to be displayed after success."* Concretely: **(a)** a successful Add Exercise must **PRESERVE the PRE-ADD values** of weight, sets, RIR, RPE, min-rep, and max-rep; **(b)** the post-success estimate/reset path must **NOT** replace those values; **(c)** a later **deliberate** exercise selection may still apply that newly selected exercise's recommendation through the existing behavior (unchanged). This resolves the two-way ambiguity raised by `automation-qa` (agent `afd85d38069c21037`) while authoring the Step-0 acceptance tests: **Reading B — "persist whatever is displayed post-success" — is REJECTED.** Consequence for the plan: `resetFormFields()` / the post-success estimate path is now **IN SCOPE** for implementation, because today it overwrites the user's values with defaults/estimates after a successful add (see Plan v2 Scope / Artifacts / Sequence).
4. Given the user triggers Clear Plan / Clear All, when the action completes, then the stored control values are cleared and the template defaults are immediately restored. *(Clearing policy — clear + reset to defaults.)*
5. Given the user triggers Clear Filters, when the action completes, then the stored control values are retained (filters are unrelated to Workout Controls). *(Clearing policy — retain.)*
6. Given the user closes the tab, when the tab is closed, then the stored control values are cleared naturally via sessionStorage expiry. *(Storage — tab-scoped sessionStorage, no cross-session persistence.)*
7. Given a single global set of Workout Controls per tab, when the user switches routine, then the same control values are retained across the routine cascade (consistent with the existing routine-cascade contract). *(Scope — one global set per tab, not per routine.)*
8. Given a valid saved value exists for a field on reload, when restore runs, then the saved value wins over the server-rendered template default. *(Conflict precedence — saved wins.)*
9. Given a stored value is missing, malformed, non-numeric, or out of range, when restore runs, then that stored value is ignored and the current template default is used for that field. *(Conflict precedence — invalid stored values fall back to defaults.)*
10. The existing contract test in `e2e/ui-hardening.spec.ts` is updated to assert the new restore-on-reload behavior, and the KI-005 row in `docs/UI_SCENARIOS_GAP_ANALYSIS.md` §0 is updated to describe the new behavior (see migration note).

**Open risks flagged for the reviewers**

1. **Restore vs. profile-estimate ordering (Issue #5) — highest-risk integration point.** `updateExerciseForm()` (lines 164-208) and `applyUserProfileEstimateForSelectedExercise()` overwrite the fields on exercise selection, and the `weightUserDirty` flag governs weight. Restore-on-reload must interleave so it is not immediately clobbered, and weight restore must respect (or participate in) the existing dirty-tracking contract.
2. **Definition of "current template default" for the criterion-9 fallback.** `initializeDefaultValues()` hardcodes defaults (weight 25 / sets 3 / rir 3 / rpe 7 / min 6 / max 8), but `updateExerciseForm()` substitutes per-exercise recommended values. The fallback target on invalid stored values needs a crisp definition (hardcoded init default vs. per-exercise recommendation).
3. **Save trigger granularity for "mid-entry".** The existing field wiring uses `change` (commit/blur) deliberately, not `input` — a refresh after typing but before blur would lose the un-committed keystrokes. Product-risk question: is `change` sufficient for the criterion-1 "mid-entry" intent, or is `input` required?
4. **New module vs. inline helper.** Introducing `workout-controls-persistence.js` vs. inlining into `workout-plan.js` — architecture-reviewer call on the cleaner seam.
5. **Retain-on-Add exact snapshot (criterion 3).** Post-success `resetFormFields()` re-applies the profile estimate, which can move the weight value; the "retained" stored set must match what is actually displayed post-success (no divergence between DOM and `sessionStorage`).
6. **sessionStorage robustness.** Key naming/collision, quota, and private-mode / disabled-storage failure handling should degrade gracefully to current (reset-on-reload) behavior rather than throwing.

### owner (Gate 1, 2026-07-12) — additional finding raised while ruling

*Not a council reviewer. Recorded here so the finding text sits alongside the reviewer findings it constrains; disposition is in the response matrix below.*

- OWNER-1 (BLOCKING — ordering hazard): Save/restore ordering must be guarded, or persistence will destroy the very state it is meant to protect. Four constraints:
  1. Initial default population must **NOT** overwrite an existing `sessionStorage` record before restore runs.
  2. Restore must occur under an **initialization/hydration suppression guard**, before persistence listeners or programmatic-save hooks can replace the saved state.
  3. Programmatic saves should occur **ONLY after hydration**, and only when a programmatic update represents the intended displayed state.
  4. Clear Plan must reset the DOM **under suppression** and remove the storage key **LAST**, leaving the key **ABSENT** rather than immediately re-saving defaults.
